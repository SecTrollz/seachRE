Node Join: QR Code Bootstrap Without a Central Token Issuer

A new node joining the mesh requires no central authority—only a single existing node's cryptographic identity and network coordinates.

Bootstrap Node Generation

The first node in the network (or any existing node acting as an introducer) generates a QR code containing:

  • Its SPKI SHA-256 pin (the public key fingerprint of its self-signed TLS certificate)
  • Its IP address and gRPC port
  • A one-time bootstrap nonce (valid for 60 seconds)
  • A network identifier hash (derived from the initial node's public key, ensuring the new node joins the correct mesh)
New Node Bootstrapping

The new node, on first boot, prompts the user to scan this QR code. The node extracts the bootstrap peer's SPKI pin and address, then:

  • Establishes a pinned TLS connection to the bootstrap peer—the certificate must match the SPKI pin exactly, eliminating the TOFU first-contact vulnerability
  • Sends a JoinRequest containing its own self-signed certificate and SPKI pin
  • The bootstrap peer adds the new node to its local peer list and gossips the addition to the mesh
  • The new node receives the full CRDT state (ad-block rules, node list) and begins participating in gossip
Zero Trust Introducers

Any node can serve as a bootstrap introducer. There is no privileged "manager" role. The trust anchor is the pinned SPKI hash of the bootstrap peer, not a central CA.

Shadow Data — What Will Go Wrong
QR Code Scanning Failures
Users will point their phone camera at the QR code and get nothing because the terminal font is too small, the screen brightness is too low, or the contrast is insufficient. Fallback: Manual entry of IP:PORT and SPKI fingerprint.
Bootstrap Peer Unavailability
If the bootstrap peer is offline during the join window, the new node hangs indefinitely. Mitigation: Display a clear "bootstrap peer unreachable—verify the node is online" message and allow scanning a different existing node's QR code.
Network Partition During Join
If the new node is on a different VLAN or behind a firewall that blocks the gRPC port, it cannot reach the bootstrap peer even though mDNS works. Fallback: Manual IP entry with port forwarding guidance.
Clock Skew
The 60-second nonce relies on reasonably synchronized clocks. A new node with a wildly incorrect system clock (e.g., fresh Raspberry Pi) will reject the bootstrap offer. Mitigation: Allow a wider tolerance (±5 minutes) and display a warning.

Ad-Block Rules Synchronization: CRDT Without a Central Database

Rules are stored as a Conflict-Free Replicated Data Type (CRDT) set that syncs across all nodes via the gossip mesh. There is no primary database, no leader, no quorum.

Data Model Each rule is a tuple (domain, action, timestamp, node_id, operation_id). Action is an enum: BLOCK, PIXEL, or CUSTOM_HTML.
CRDT Type Observed-Remove Set (OR-Set) where each element has a unique identifier and a vector clock. Deletions are represented as tombstones with the same identifier.
Sync Protocol When a node adds or removes a rule, it broadcasts the operation via gossip. Peers apply the operation deterministically: later timestamp wins; if timestamps are equal, higher node_id wins.
Implementation Use the automerge crate—a production-grade CRDT library with Rust + WebAssembly bindings. automerge-repo handles the network synchronization layer.
·Every node maintains the full rule set locally in SQLite (for fast querying by the DNS server) while the CRDT document serves as the source of truth for sync.
·A node that was offline for a week catches up by requesting operations since its last known vector clock from any gossip peer.
·Conflicting rules (e.g., two users adding different actions for the same domain) are resolved automatically; the "winning" rule is active, but the history is preserved.
Shadow Data — What Will Go Wrong
CRDT Document Bloat
With thousands of rules and frequent changes, the CRDT document grows linearly. Tombstones are never deleted (by design, for causality). After a year of operation, sync may become slow. Mitigation: Implement garbage collection that prunes tombstones older than 30 days, assuming all nodes have seen them.
Divergent States
If a node is partitioned for months and misses garbage collection, it may have a different view of which tombstones are safe to delete. Mitigation: Nodes gossip their vector clocks and refuse to GC until all active peers have acknowledged the tombstone.
Large Initial Sync
A new node joining a long-running mesh must download the entire CRDT history. For 10,000 rules this is manageable (a few MB). For 1,000,000 rules it's a problem. Mitigation: Snapshot mechanism—a node exports its current state as a compact binary and serves it over HTTP for new nodes to bootstrap from.

Dashboard Resilience: Client-Side Failover When a Node Goes Down

The dashboard is a static HTML/JavaScript bundle served by every node on its own embedded HTTP server. When the user visits http://<any-node-ip>:8080, they get the same dashboard, and the dashboard does not rely on that node staying up.

  1. Peer List Discovery
    On load, the dashboard queries the local node's /peers endpoint, which returns the current list of known mesh nodes (IPs, ports, SPKI pins) from the node's local CRDT state.
  2. Parallel Hedged Requests
    For every gRPC call (e.g., GetClusterStatus), the dashboard selects three random nodes from the peer list and sends the request to all three simultaneously. The first successful response is used; the others are canceled. This mitigates node failures transparently.
  3. Multi-Node WebSocket Fallback
    For streaming data (e.g., live DNS query logs), the dashboard opens WebSocket connections to multiple nodes. Each event includes a Lamport timestamp and node_id. The dashboard deduplicates events by keeping a sliding window of seen (node_id, timestamp) tuples. If one WebSocket dies, the others continue feeding data.
  4. Local Cache of Peer List
    The dashboard stores the last known peer list in localStorage. If the initial node is unreachable, it attempts to connect to cached peers directly.
Shadow Data — What Will Go Wrong
CORS and Mixed Content
The dashboard loaded from http://node1:8080 attempts to make gRPC-Web requests to https://node2:8443. Browsers block this due to CORS unless the node's HTTP server includes Access-Control-Allow-Origin headers. Mitigation: Every node's embedded server includes Access-Control-Allow-Origin: * (acceptable for LAN-only dashboards) and serves the dashboard over HTTPS only, with a self-signed cert.
Browser Certificate Errors
Each node uses a self-signed certificate. The dashboard, when making requests to multiple nodes, will encounter certificate warnings for each new node's IP. Mitigation: The user must manually accept each node's certificate on first connection (browser-native TOFU), or the dashboard is served over plain HTTP for LAN-only use with a clear warning.
WebSocket Message Flood
With 10 nodes and 3 WebSocket connections, the dashboard receives duplicate events for every DNS query. Mitigation: Subscribe to only one WebSocket at a time, with a failover mechanism that reconnects to the next node on disconnect.

mTLS Without a Central CA: TOFU + SPKI Pinning

The system uses mutual TLS (mTLS) between nodes without any central certificate authority. Trust is established via Trust On First Use (TOFU) hardened with SPKI pinning.

1 · Self-Signed Certificates
Every node generates a self-signed X.509 certificate on first boot. The certificate includes a unique node_id in the Subject Alternative Name.
2 · SPKI Pin Calculation
Each node computes the SHA-256 hash of its certificate's Subject Public Key Info (SPKI) and logs it at startup. This pin is the node's cryptographic identity.
3 · Pinned First Contact
When Node A connects to Node B for the first time (via QR code bootstrap), Node A has been provided with Node B's SPKI pin from the QR code. The TLS handshake verifies that the presented certificate's SPKI hash matches the expected pin exactly. This eliminates the TOFU first-contact MITM vulnerability.
4 · Subsequent Connections
Once a connection succeeds, Node A stores Node B's (node_id, SPKI_pin, certificate) in a local trust store. Future connections verify the pin matches the stored value. If the pin changes (e.g., due to certificate regeneration), the connection is rejected and the user is alerted.
5 · Mesh Propagation
When a new node joins via bootstrap, the bootstrap peer gossips the new node's node_id and SPKI pin to the rest of the mesh. Other nodes can now connect to the new node with pinned trust, without ever having performed a TOFU handshake directly.

The trust anchor is the bootstrap QR code, exchanged out-of-band by physically scanning a screen. Analogous to SSH host key verification. SPKI pinning is immune to certificate regeneration—the pin depends only on the public key, which can be reused across certificate renewals.

Shadow Data — What Will Go Wrong
User Ignorance
Users will scan the QR code without understanding it contains a cryptographic trust anchor. 99% will not verify the SPKI fingerprint out-of-band, same as SSH. Mitigation: The QR code screen displays a prominent warning: "Only scan this if you are physically present with the node. Verify the fingerprint matches the label on the device."
Certificate Regeneration
If a node's disk fails and it regenerates a new certificate with a new public key, its SPKI pin changes and all other nodes will reject connections. Mitigation: Provide a "reset node identity" workflow that requires manual intervention (e.g., physically pressing a button on the device) and broadcasts a revocation message.
Key Compromise
If a node's private key is stolen, the attacker can impersonate that node. There is no central revocation authority. Mitigation: Nodes can broadcast a NodeRevoked message signed by the compromised node's private key (if available) or rely on manual removal from the mesh via the dashboard.
Pro (TOFU) Con (TOFU)
Zero infrastructure overhead—no CA to deploy, secure, or maintain. First-contact vulnerability if the bootstrap pin is not verified out-of-band.
No single point of trust compromise—compromising one node's key does not compromise the entire mesh. No centralized revocation—revoking a compromised node requires manual propagation or a gossip-based revocation list.
Works offline—no dependency on external OCSP/CRL servers. User friction—certificate changes trigger warnings that users must manually accept.
Scales horizontally—trust is peer-to-peer, not bottlenecked through a CA. No cross-mesh trust—nodes from different meshes cannot automatically trust each other without a shared root of trust.
Resistant to CA compromise—unlike the Web PKI, where a single rogue CA can issue certs for any domain. Difficult auditability—no central log of issued certificates; each node maintains its own trust store.

The default framing—"DePIN mining on residential hardware is not profitable"—is true and incomplete at the same time. It describes the median outcome for the median participant who joins after public attention, runs one node at home, and sells tokens immediately. That is not the only configuration. What follows is the version nobody writes down, built from the reward algorithm source code, tokenomic papers, and network economics rather than project marketing.

▸ Vector 1: Emission Curve Front-Running
The Identity Reward per node = emission_rate(t) ÷ node_count(t). Both variables change over time. Emission decays on a schedule written in the whitepaper. Node count grows as the project receives press coverage.
The Asymmetry Under exponential decay emission, block rewards are mathematically highest when participation is lowest. The reward-per-node curve peaks before public attention, not after. This is not opinion—it is the shape of the function.
The Crossover There is a date T* where the rate of node count growth exceeds the rate of emission decay. Before T*, each new node earns more than the last. After T*, each new node earns less. The profitable window is before T*, not after the first Medium article.
The Action Read the whitepaper emission schedule. Find the halving dates. Query the live explorer API for current node count. Build the crossover model. Enter before T*. The rainbowd scheduler in this system already tracks emissions and node counts per protocol—this is the missing query it needs to answer.

60% of active DePIN tokens have scheduled halving events or emission reductions within 18 months as of Q1 2025. The schedule is public. The crossover date is calculable. Most people never calculate it.

▸ Vector 2: Geographic Density Arbitrage
The Algorithm Helium's HIP-17 density scaling reduces rewards for nodes in dense hexagons. A documented example: hex with 61 hotspots, reward scale reduced from 1.000 → 0.066. That is a 15× penalty for being in a popular location.
The Inverse A node in a sparse hex earns full scale (1.000). Same hardware. Same electricity cost. 15× earnings difference from placement alone. A rural node 30 miles from the nearest hotspot earns more than a city node in a neighborhood with six others.
The Ring Pattern The adjacency bonus rewards nodes placed in neighboring hexagons (up to 6 neighbors). A 6-node ring placed one-per-hex in a coverage gap captures: full reward scale at every node + adjacency multiplier at each. A 6-node ring in sparse territory can mathematically outperform 60 nodes packed into a saturated city hex.
The Action Before deploying a node, query the coverage map API. Find hexes with reward scale ≥ 0.8 and zero adjacent hotspots. Place the first node there. Plan a ring of 6. This is what the protocol is explicitly designed to incentivize—the information asymmetry is that most people deploy at home by default.

▸ Vector 3: Type A vs Type B Protocol Filter

Two structurally different things are called "DePIN." Mixing them up is the most expensive mistake a node operator makes.

Type Revenue Source What Happens When Emissions Drop Examples
Type A · Token-Circular Miners earn tokens. Only other miners buy those tokens. Revenue is self-referential. Collapse. No external demand exists to replace emission rewards. Farm-and-dump accelerates. Most small DePIN launches. If you can't find who pays the network besides miners, it's Type A.
Type B · Usage-Backed Real external buyers (enterprises, AI firms, IoT platforms) pay fiat for services. Tokens are burned to access capacity. Node operators still earn from usage fees. Token scarcity increases as burns continue. Two revenue streams compound. Aethir ($127.8M enterprise GPU revenue, 2025). Filecoin storage fees from external users. Render GPU jobs from production studios.
The Filter Before joining any network: find the burn rate. Daily tokens burned by actual service usage, not by miners paying themselves. If burn rate ≈ 0, it is Type A. If burn rate is growing toward emission rate, it is Type B trending toward deflation.
The Entry Point Enterprise contracts are signed months before they appear in explorer dashboards. The signal is partnership announcements with named enterprises, not token price charts. Enter Type B networks when burn rate is 10–30% of emissions and growing—before the ratio becomes visible to the mainstream.
The rainbowd Connection The Thompson Sampling scheduler in this system selects protocols by profitability. The missing input is burn-rate-to-emission ratio per protocol, fetched live. A protocol with burn/emission ratio trending toward 1.0 should receive increasing alpha in the sampler regardless of current token price.

▸ Vector 5: Unstaked Token Float in Burn-and-Mint Networks
The Mechanism In burn-and-mint equilibrium networks, enterprise users burn tokens to purchase service credits. Miners who stake and hold own a growing share of a shrinking float. This is not mining income—it is a position in a scarce asset that happens to also pay yield. The two streams compound.
The Signal Watch the burn rate, not the token price. When daily_burns ÷ daily_emissions approaches 1.0, the token is approaching supply equilibrium. When burns exceed emissions, the token supply is contracting while existing staked positions represent a larger share of a smaller total. This is the second revenue stream most node operators never capture because they sell immediately.
The Threshold The burn/emission ratio is publicly readable from chain state on any Solana or EVM DePIN. The profitable configuration is: enter early (Vector 1), place correctly (Vector 2), only in Type B protocols (Vector 3), stake and hold rather than dump, and watch the ratio. Exit when either the protocol stalls or you need liquidity—not on a fixed schedule.
◆ What rainbowd Is Missing: The Alpha Engine

The Thompson Sampling scheduler already designed for this system selects protocols by trailing profitability. That is reactive. The five vectors above are predictive. The missing module—no one has shipped it as an open tool—combines them:

1Crossover Calculator: For each supported protocol, fetch the emission schedule from the whitepaper and the live node count from the explorer API. Compute T* (the date emission decay rate exceeds node growth rate). Surface "before crossover / at crossover / saturated" per protocol. This is the entry signal.
2Hex Density Scanner: For coverage-based protocols (Helium Mobile, DIMO, Hivemapper), query the coverage map API for the node's current H3 hex density and reward scale. Surface the nearest sparse hex and the optimal ring placement coordinates. This is the placement signal.
3Burn/Emission Monitor: For each protocol, compute the rolling 7-day burn/emission ratio from on-chain data. Protocols with ratio trending toward 1.0 receive increased weight in the Thompson sampler regardless of current token price. This is the hold signal.
4Protocol Type Classifier: Automatically flag protocols as Type A (token-circular) or Type B (usage-backed) based on external revenue sources found in on-chain fee data. Refuse to schedule Type A protocols when emissions are within 6 months of a halving event. This is the filter.

These four queries, running continuously and feeding the Thompson sampler's reward estimates, turn the mining scheduler from a reactive load balancer into a structural edge. The data is all public. The math is not complex. Nobody has assembled it into a single decision layer because the people who figured it out are running it quietly, and the people who write about DePIN are selling something else.

Security Trade-Offs: TOFU vs. Central CA

Model Pros Cons
Central CA Strong initial trust (certs chain to single offline root); easy rotation; centralized revocation (CRL/OCSP). Single point of failure and compromise. If the CA private key is stolen, the attacker can impersonate any node. If CA is offline, new nodes cannot join and existing nodes cannot renew certs. The CA is the spine we were trying to avoid.
TOFU + SPKI Pinning No single point of failure. Each node's trust is anchored to its own initial bootstrap interaction. Compromising one node does not compromise the mesh. Works offline. Scales infinitely. First-contact vulnerability if the bootstrap pin is not verified out-of-band. No central revocation. Certificate changes require manual user acceptance.

The bootyloversdelight choice: TOFU + SPKI Pinning. The threat model is residential networks, not nation-state adversaries. The risk of a MITM during the first 60-second QR code window is acceptable given the physical presence requirement. The resilience benefit of eliminating the CA single point of failure outweighs the theoretical first-contact vulnerability.

The 3 AM Sysadmin Summary

Every box runs the exact same binary. You unbox a new node, plug it in, and it shows a QR code on its little screen. You point your phone at it, scan, and the new node joins the mesh automatically. All your ad-block rules sync across every node without you touching a database. If one node catches fire, the dashboard on your phone just talks to the next closest node and you don't even notice. No cloud, no master server, no "please contact your administrator" errors. It's a swarm of identical bees. You can unplug half of them and the hive keeps making honey. Now go get some sleep—the network's got this.
